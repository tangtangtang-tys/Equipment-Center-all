# 设备详情页阶段目标需求梳理 PRD

## 1. Executive Summary

### 1.1 Problem Statement

当前设备中心详情页已经具备设备查询、基础信息、绑定用户、增值服务、日志、设备状态、物联网卡和禁用能力雏形，但整体仍偏“字段展示”。客服和运维在处理单设备问题时，仍需要跨模块、跨系统、跨接口人工判断，且字段空态、数据来源、设备生命周期节点、禁用影响、日志证据之间缺少统一表达。

旧系统还存在一个关键历史问题：大量业务判断由前端基于设备类型和多个底层接口自行拼接。例如低功耗 4G 设备需要同时读取低功耗、物联网卡、增值服务、报警日志等接口。该模式导致重构难、口径不一致、客服解释依赖经验、研发排查依赖旧代码反推。

### 1.2 Proposed Solution

下阶段不重构为强诊断工作台，也不打散现有 Tab 结构，而是将设备详情页升级为“信息标准化 + 设备履历 + 轻量辅助排查”的详情页：

- 保留现有 Tab：添加该设备的用户、设备状态、增值服务、日志记录、物联网卡。
- 新增“设备履历 / 生命周期”Tab，承载资产、使用、安全三条生命周期。
- 增加“辅助诊断”入口，以抽屉形式提供检查清单、需关注项、缺失项和下一步查看入口。
- 禁用不作为独立模块，而是嵌入基础信息、绑定用户、增值服务、日志记录和操作日志。
- 统一字段空态、状态枚举、模块边界和交互入口。

### 1.3 Success Criteria

建议以下指标作为阶段验收口径，具体数值可在上线前结合埋点能力确认：

| 指标 | P0 目标 | 说明 |
| --- | --- | --- |
| 关键字段覆盖率 | 设备详情 P0 字段覆盖率 >= 90% | P0 字段包括基础信息、用户、状态、服务、日志、物联网卡核心字段 |
| 字段空态规范覆盖率 | 100% P0 字段支持统一空态 | 区分暂无数据、暂未接入、不适用、获取失败、未知 |
| 单设备基础判断效率 | 客服/运维可在 1 个详情页内完成基础判断 | 是否存在、是否激活、是否绑定、运行状态、可用状态、服务大区 |
| 模块边界一致性 | 现有模块不被打散，新增模块不影响原流程 | 新增设备履历 Tab 与辅助诊断入口不替代原有 Tab |
| 高风险操作追溯 | 禁用/恢复记录 100% 进入操作日志 | 包含时间、操作人、前后状态、原因、说明 |

## 2. User Experience & Functionality

### 2.1 User Personas

| 用户角色 | 当前阶段关注点 | 主要使用模块 |
| --- | --- | --- |
| 客服 / 售后 | 快速确认用户设备能不能用、是否绑定、是否被禁用、服务是否有效 | 顶部摘要、绑定用户、增值服务、日志记录、辅助诊断 |
| 运维 / 技术支持 | 查看在线状态、心跳、P2P、MQTT/TCP、节点、大区、日志证据 | 设备状态、日志记录、物联网卡、辅助诊断 |
| 超管 / 风控 | 查看完整状态、执行禁用/恢复、查看操作日志 | 基础信息、操作日志、禁用/恢复入口 |
| 运营 / 产品 | 查看设备基础事实、服务状态、问题分布的基础数据 | 基础信息、增值服务、设备履历 |
| 后续供应链 / 质量 / 财务 | 查看资产流转、质检、销售、维修、报废等信息 | 设备履历 / 生命周期 |

### 2.2 User Stories

#### Story 1：查看设备基础全貌

As a 客服/运维, I want to see a concise device summary and grouped basic information, so that I can quickly confirm what this device is, who it belongs to, and whether it has basic usage prerequisites.

Acceptance Criteria:

- 顶部展示设备 ID、型号、产品线/连接类型、激活状态、绑定状态、运行状态、设备可用状态、服务大区、最近心跳。
- 基础信息按身份信息、产品与能力、生产信息、激活与归属、可用与限制状态分组。
- 字段缺失时必须使用统一空态，不允许只展示 `--`。
- 禁用状态展示在“可用与限制状态”中，不新增独立禁用 Tab。

#### Story 2：查看设备绑定用户

As a 客服, I want to see the current bound users and their roles, so that I can determine whether the reporting user has permission to use this device.

Acceptance Criteria:

- 展示绑定状态、主账号数量、共享账号数量、用户 ID、用户角色、添加时间、推送状态。
- 支持跳转用户中心。
- 设备禁用时，在用户模块展示禁用对绑定用户的影响说明。
- P1 支持绑定/解绑历史和用户核对能力。

#### Story 3：查看设备运行与连接状态

As an 运维, I want to see online state, heartbeat, P2P, MQTT/TCP, network, low power, firmware status, so that I can verify whether the device is capable of real-time connection.

Acceptance Criteria:

- 展示在线状态、最近心跳、最近连接、最近断开、连接类型、连接节点、P2P UID、P2P 状态、IP、网络类型、电量、低功耗状态、当前固件版本、拉黑状态。
- 低功耗设备展示提示：休眠不等于异常离线，需结合心跳、唤醒、报警日志判断。
- 按设备能力动态展示字段，非 4G 设备不展示 4G 卡相关状态，非车载设备不展示速度/经纬度。

#### Story 4：查看增值服务状态

As a 客服/售后, I want to see cloud storage, AI, 4G package and other service statuses, so that I can answer whether the user has valid service rights.

Acceptance Criteria:

- 展示服务名称、服务类型、开通状态、有效期。
- 生效中服务优先展示。
- 设备禁用时展示服务影响说明：不自动退款、延期、补偿、权益转移；历史云录像仍可查看。
- P1 支持订单、购买用户、自动续费、绑定设备、绑定卡号、AI 授权、4G 套餐状态。

#### Story 5：查看日志证据

As an 运维/客服, I want to view login, reset, alarm, low-power and operation logs, so that I can trace what happened to the device.

Acceptance Criteria:

- 日志总览展示最近登录、最近报警、最近重置、今日登录次数、今日报警次数、当前查询大区。
- 登录日志展示时间、状态、IP、版本。
- 重置记录展示重置时间。
- 报警日志展示 UTC 时间、北京时间、动作、电量、元数据和资源文件入口。
- 禁用、恢复等人为操作进入操作日志。

#### Story 6：查看物联网卡信息

As a 客服/运维, I want to view the current SIM card and package status for 4G devices, so that I can verify whether cellular connectivity may affect device usage.

Acceptance Criteria:

- 4G 设备展示当前 ICCID、IMEI、插卡位置、卡类型、插卡记录、机卡绑定是否支持。
- 非 4G 设备展示“不适用”。
- P1 支持白名单、套餐名称、套餐状态、套餐有效期、订单卡号匹配。

#### Story 7：查看设备履历 / 生命周期

As a 客服/运维/后续质量人员, I want to see the device lifecycle history, so that I can understand the key stages this device has gone through.

Acceptance Criteria:

- 新增“设备履历 / 生命周期”Tab。
- 展示当前阶段摘要、资产生命周期、使用生命周期、安全生命周期、关键事件时间线。
- 第一版允许部分节点标注“暂未接入”，但必须明确区分未接入、未发生、信息缺失。
- 不将生命周期堆叠在顶部摘要区。

#### Story 8：使用辅助诊断

As a 客服/运维, I want an auxiliary diagnostic drawer, so that I can know which modules to check next without relying purely on experience.

Acceptance Criteria:

- 辅助诊断不作为独立 Tab。
- 第一版覆盖设备离线/无法预览、用户看不到设备、云存储无录像/有报警无文件、4G 设备不上线、设备被禁用。
- 诊断结果只使用正常、需关注、信息不足、不适用。
- 不输出设备故障、责任归属、自动修复结论。
- 输出检查项、依据、缺失数据、建议查看入口。

### 2.3 Non-Goals

当前阶段不做：

- 不重构为强诊断工作台。
- 不打散现有 Tab 结构。
- 不将禁用设计为独立模块。
- 不做批量禁用、批量恢复、复杂审批流。
- 不自动退款、延期、补偿、权益转移。
- 不自动生成工单或自动修复设备。
- 不对设备故障、用户责任、渠道责任做系统定责。
- 不在 P0 阶段强依赖资产流转、质检、维修、报废、OTA、配置下发等未接入系统。

## 3. AI System Requirements

当前需求不涉及 AI 自动决策系统。

如果未来引入智能诊断或智能客服建议，需要另行定义：

- 输入数据范围。
- 诊断规则和模型责任边界。
- 结果可解释性。
- 人工复核流程。
- 误判评估集和准确率指标。

## 4. Technical Specifications

### 4.1 Architecture Overview

设备详情页数据建议按“事实聚合 + 模块展示 + 辅助排查”组织：

```text
设备 ID 查询
  ↓
设备中心聚合服务
  ├─ 设备基础信息
  ├─ 实时状态 / 主子节点
  ├─ 用户绑定
  ├─ 增值服务
  ├─ 日志系统
  ├─ 物联网卡系统
  ├─ 操作日志
  └─ 生命周期事件
  ↓
设备详情页
  ├─ 顶部摘要
  ├─ 基础信息区
  ├─ 原有 Tab
  ├─ 设备履历 Tab
  └─ 辅助诊断抽屉
```

### 4.2 Integration Points

| 数据域 | 可能来源 | 当前阶段要求 |
| --- | --- | --- |
| 设备基础信息 | 设备中心接口、生产数据、产品配置 | P0 |
| 实时状态 | 主节点 / 子节点 Redis、MQTT/TCP 状态 | P0 |
| 用户绑定 | 用户中心 / 绑定关系服务 | P0 |
| 增值服务 | 订单系统、权益系统、云存储、AI、4G 套餐 | P0 展示基础状态，P1 补订单与配置 |
| 日志记录 | 登录日志、报警日志、重置记录、低功耗日志 | P0 展示登录/报警/重置，P1 补低功耗 |
| 物联网卡 | SIM 卡服务、套餐服务、白名单服务 | P0 展示基础卡信息，P1 补状态与套餐 |
| 操作日志 | 设备中心审计日志 | P0 禁用/恢复记录，P1 统一高风险操作 |
| 生命周期 | 基础信息、用户、状态、日志、供应链/售后系统 | P0 轻履历，P2 深度资产流转 |

### 4.3 Page Structure

页面结构：

```text
设备详情页
├─ 顶部设备摘要
├─ 基础信息区
├─ 辅助入口
│  ├─ 辅助诊断
│  └─ 操作日志
└─ 详情 Tab
   ├─ 添加该设备的用户
   ├─ 设备状态
   ├─ 增值服务
   ├─ 日志记录
   ├─ 物联网卡
   └─ 设备履历 / 生命周期
```

### 4.4 Interaction Logic

#### 4.4.1 查询与进入详情

```text
用户输入设备 ID
  ↓
系统查询设备基础信息
  ↓
设备存在：进入详情页
设备不存在：展示不存在 / 未同步提示
  ↓
详情页默认展示顶部摘要、基础信息、第一个 Tab
```

交互要求：

- 查询为空时提示“请输入设备 ID”。
- 设备不存在时提示“设备不存在或数据未同步”。
- 接口失败时提示“获取失败”，而不是“暂无数据”。

#### 4.4.2 Tab 切换

- 保留现有 Tab 切换方式。
- Tab 内容各自负责自身模块，不跨模块堆叠信息。
- 需要跨模块查看时使用提示和跳转入口。

#### 4.4.3 设备履历 / 生命周期

- 用户点击“设备履历 / 生命周期”Tab 后进入履历视图。
- 顶部展示当前阶段摘要。
- 中部展示三条生命周期。
- 底部展示关键事件时间线。
- 未接入节点必须明确标注“暂未接入”。

#### 4.4.4 辅助诊断

```text
用户点击辅助诊断
  ↓
打开右侧抽屉
  ↓
选择问题类型或使用通用检查
  ↓
系统读取当前详情页已接入数据
  ↓
展示检查结果摘要、检查项、缺失数据、建议查看入口
```

辅助诊断结果规则：

- 只提示需关注项，不定责。
- 每个检查项必须显示依据字段。
- 数据缺失时展示信息不足。
- 非当前设备类型适用项展示不适用。

#### 4.4.5 禁用 / 恢复

- 禁用状态展示在基础信息的“可用与限制状态”中。
- 禁用/恢复按钮放在设备可用状态旁或管理设备区。
- 无权限角色只可查看状态，不展示或置灰操作按钮。
- 执行禁用/恢复时必须填写原因和说明。
- 禁用/恢复后写入操作日志。

#### 4.4.6 操作日志

- 顶部提供操作日志快捷入口。
- 日志记录 Tab 内也保留操作日志入口。
- 禁用、恢复、编辑服务大区、限制推送、解绑 UID 等后续统一进入操作日志。

### 4.5 Module Requirements

#### 4.5.1 顶部设备摘要

字段：

- 设备 ID。
- 设备型号。
- 产品线 / 连接类型。
- 激活状态。
- 绑定状态。
- 运行状态。
- 设备可用状态。
- 服务大区。
- 最近心跳。

#### 4.5.2 基础信息区

分组：

- 身份信息：设备 ID、SN、UUID、IMEI、ICCID、MAC。
- 产品与能力：产品线、设备型号、机型型号、连接类型、网络类型、电量类型、是否低功耗、是否支持 TF 卡。
- 生产信息：生产工厂、生产时间、生产批次、出厂固件。
- 激活与归属：激活状态、激活时间、首次上线时间、归属组织、经销商、服务大区、所在地。
- 可用与限制状态：设备可用状态、拉黑状态、影子拉黑、移动推送受限、最近状态变更。

#### 4.5.3 添加该设备的用户

P0 字段：

- 绑定状态。
- 主账号数量。
- 共享账号数量。
- 用户 ID。
- 用户角色。
- 添加时间。
- 推送状态。
- 用户中心跳转。

P1 字段：

- 手机号 / 邮箱脱敏。
- 用户昵称。
- 共享权限。
- 用户核对。
- 绑定 / 解绑历史。

#### 4.5.4 设备状态

P0 字段：

- 在线状态。
- 最近心跳。
- 最近连接时间。
- 最近断开时间。
- 连接类型。
- 连接节点 / 子节点。
- P2P UID。
- P2P 状态。
- IP / 地区。
- 网络类型。
- 电量。
- 低功耗状态。
- 当前固件版本。
- 设备拉黑状态。
- 影子拉黑状态。

P1 字段：

- P2P 模式。
- 最近推流时间。
- 穿透类型。
- UID 分配 / 回收原因。
- 供电方式。
- 唤醒原因。
- 最近 7 天活跃。
- OTA 状态。

#### 4.5.5 增值服务

P0 字段：

- 服务名称。
- 服务类型。
- 开通状态。
- 有效期。
- 生效中服务优先展示。
- 禁用影响提示。

P1 字段：

- 购买用户。
- 订单号。
- 自动续费状态。
- 绑定设备。
- 绑定卡号。
- 4G 套餐状态。
- AI 授权状态。

P2 字段：

- 影子配置状态。
- PMS 下发状态。
- 缓存刷新状态。
- 配置失败原因。

#### 4.5.6 日志记录

P0 字段：

- 日志总览：最近登录时间、最近报警时间、最近重置时间、今日登录次数、今日报警次数、当前查询大区。
- 登录日志：登录时间、登录状态、IP、版本。
- 重置记录：重置时间。
- 报警日志：UTC 时间、北京时间、动作、电量、元数据、资源文件。

P1 字段：

- 低功耗日志。
- 调试日志。
- 资源文件抽屉。
- UTC / 北京时间切换。
- 报警动作筛选。
- 有无资源筛选。

#### 4.5.7 物联网卡

P0 字段：

- 是否 4G 设备。
- 当前 ICCID。
- IMEI。
- 插卡位置。
- 卡类型。
- 插卡记录。
- 机卡绑定是否支持。
- 卡号脱敏。

P1 字段：

- 当前卡识别。
- 首次 / 最近插卡时间。
- 白名单状态。
- 卡来源。
- 套餐名称。
- 套餐状态。
- 套餐有效期。
- 订单卡号匹配。

#### 4.5.8 设备履历 / 生命周期

信息层级：

```text
当前阶段摘要
资产生命周期
使用生命周期
安全生命周期
关键事件时间线
```

资产生命周期：

```text
生产 → 质检 → 入库 → 出库 → 销售 → 退货 → 维修 → 翻新 → 报废
```

使用生命周期：

```text
未激活 → 已激活 → 已绑定 → 已安装 → 运行中 / 休眠 → 离线 → 停用 → 解绑 → 转移 → 退役
```

安全生命周期：

```text
出厂固件 → 首次联网 → 配置下发 → OTA 升级 → 告警 → 故障 → 远程诊断 → 修复
```

关键事件时间线：

- 生产。
- 激活。
- 绑定。
- 登录。
- 报警。
- 重置。
- 禁用。
- 恢复。
- OTA。
- 维修。

生命周期节点状态：

| 状态 | 含义 |
| --- | --- |
| 已完成 | 有明确时间或记录 |
| 当前阶段 | 当前设备正处于该状态 |
| 未发生 | 确认还没有发生 |
| 暂未接入 | 当前系统尚未接入该数据源 |
| 信息缺失 | 理论应有，但接口未返回 |
| 不适用 | 当前设备类型不适用 |

### 4.6 Security & Privacy

| 数据 | 要求 |
| --- | --- |
| 手机号 / 邮箱 | 默认脱敏，完整信息需权限 |
| ICCID / IMEI | 默认可按角色脱敏，完整信息需权限 |
| IP 地址 | 客服可视情况脱敏，运维可查看完整 |
| 禁用原因 | 经销商和低权限角色不展示敏感内部原因 |
| 原始元数据 | 仅运维、研发、超管等角色可查看 |
| 操作日志 | 普通角色只看摘要，高权限角色看完整原因和操作者 |

权限设计应基于“角色 + 权限点 + 数据范围”，不应只依赖前端硬编码角色。

### 4.7 Testing & Acceptance

P0 验收建议：

- 使用 IPC、NVR、车载、BK、4G 低功耗至少各 1 台样例设备验证详情页字段展示。
- 使用已激活、未激活、已绑定、未绑定、在线、离线、休眠、禁用、非 4G、4G 有卡等设备状态覆盖测试。
- 验证所有 P0 字段空态是否符合规范。
- 验证无权限角色不可执行禁用/恢复。
- 验证禁用/恢复记录写入操作日志。
- 验证设备履历 Tab 中未接入节点展示为“暂未接入”，不展示为普通空值。
- 验证辅助诊断不输出强结论，只输出检查项和建议入口。

## 5. Risks & Roadmap

### 5.1 Current Problems To Sync Into Planning

当前阶段需要同步纳入规划的问题：

1. 旧系统业务判断大量依赖前端拼接口和旧代码反推。
2. 不同产品线设备字段口径不统一，IPC、NVR、车载、BK、4G 低功耗差异较大。
3. 当前详情页已有基础结构，但信息分组、字段空态、数据来源、权限边界仍不够清晰。
4. 增值服务、报警日志、低功耗、物联网卡之间存在联动关系，但当前阶段不适合做强自动诊断。
5. 禁用能力已经被定义为高风险操作，不应仅作为普通状态按钮，也不宜单独成为详情页一级模块。
6. 生命周期有价值，但不应堆叠在顶部，适合独立设备履历 Tab 承载。
7. 客服和运维是当前阶段核心用户，后续需要兼容供应链、质量、财务、风控、经销商等角色。

### 5.2 Phased Rollout

#### P0：信息标准化阶段

阶段目标：

```text
完成设备详情页基础信息标准化，保证客服/运维能在一个页面内看清设备核心事实。
```

交付内容：

- 顶部设备摘要。
- 基础信息分组。
- 添加该设备的用户模块补齐。
- 设备状态模块补齐。
- 增值服务基础状态。
- 日志基础结构。
- 物联网卡基础信息。
- 设备履历 / 生命周期 Tab 第一版。
- 禁用状态嵌入基础信息与操作日志。
- 字段空态规范落地。

验收重点：

- P0 字段覆盖率 >= 90%。
- P0 字段 100% 支持统一空态。
- 设备履历 Tab 可展示已有节点和未接入节点。
- 禁用不作为独立 Tab。

不做：

- 不做强自动诊断。
- 不做复杂审批流。
- 不做资产全链路流转。
- 不做自动修复。

#### P1：辅助排查增强阶段

阶段目标：

```text
在信息标准化基础上，补充辅助诊断、关键历史记录、服务订单和资源文件查看能力。
```

交付内容：

- 辅助诊断抽屉。
- 绑定历史。
- 服务订单关联。
- 资源文件抽屉。
- 低功耗日志。
- 白名单 / 套餐状态。
- 生命周期事件时间线增强。
- 操作日志覆盖更多高风险操作。

验收重点：

- 辅助诊断覆盖 5 类常见问题。
- 辅助诊断每项均有依据字段、缺失数据和建议入口。
- 资源文件可在报警日志中查看。
- 低功耗设备能展示唤醒、报警、上传等关键事件。

不做：

- 不输出故障定责。
- 不自动生成工单。
- 不自动执行禁用/恢复。

#### P2：深度治理阶段

阶段目标：

```text
接入更多生命周期数据和审计能力，支撑售后、风控、质量、供应链等角色进一步使用设备中心。
```

交付内容：

- 资产流转接入：质检、入库、出库、销售。
- OTA / 配置下发接入。
- 维修 / 翻新 / 报废接入。
- 操作审计统一化。
- 角色视图配置。
- 设备履历跨系统单据跳转。

验收重点：

- 资产生命周期节点不再只停留在生产信息。
- 操作日志覆盖主要高风险操作。
- 不同角色可基于权限看到不同字段和入口。

不做：

- 不直接替代供应链、财务、质量、售后系统。
- 不在设备中心内完成复杂财务结算或售后赔付。

#### P3：主动预警与规模化分析阶段

阶段目标：

```text
从单设备详情查询演进到批量异常发现、主动预警和运营分析。
```

交付内容：

- 批量异常设备列表。
- 健康评分或风险标签。
- 固件版本异常分析。
- 大区 / 节点 / 型号异常分布。
- 设备地图与运营看板增强。
- 主动预警规则。

验收重点：

- 可按设备类型、固件版本、大区、服务状态筛选异常设备。
- 可对连续离线、低电量、P2P 异常、服务异常等规则生成预警。

不做：

- 不在 P3 前提早建设不可信的大屏。
- 不在数据口径未稳定前做自动治理闭环。

### 5.3 Technical Risks

| 风险 | 说明 | 应对 |
| --- | --- | --- |
| 数据源不完整 | 资产、套餐、白名单、OTA、配置下发等可能未接入 | 明确展示暂未接入，避免伪装为空 |
| 设备类型差异大 | IPC、NVR、车载、BK、4G 低功耗字段不同 | 按设备能力动态展示 |
| 前端继续承载复杂判断 | 可能重复旧系统问题 | 将判断规则产品化，后续逐步后端聚合 |
| 辅助诊断被误解为最终结论 | 客服可能直接按提示定责 | 文案明确“仅供排查参考”，不输出强结论 |
| 禁用影响被误解 | 禁用可能被误认为自动解绑或自动退款 | 在用户、服务、日志中展示影响边界 |
| 权限泄露 | 原始日志、禁用原因、用户信息、卡号等敏感 | 按权限点和数据范围控制 |

### 5.4 Open Questions

以下问题建议后续继续确认：

1. P0 阶段各业务线设备样例是否已准备好，用于字段和空态验收。
2. 用户手机号、ICCID、IMEI、IP 的脱敏规则是否有统一规范。
3. 设备履历 Tab 中资产生命周期的质检、入库、出库、销售数据是否已有权威系统。
4. 辅助诊断第一版是否只开放给内部客服/运维，不开放给经销商。
5. 操作日志是否由设备中心统一落库，还是先聚合各系统操作记录。
6. P0 上线时间、研发排期、接口依赖方仍需项目排期确认。

## 6. Summary

本阶段设备详情页的核心不是“做更多页面”，而是“把现有详情页做标准、做清楚、做可解释”。

推荐方向：

```text
设备详情页先做好信息标准化；
生命周期作为新增 Tab 承载设备履历；
辅助诊断作为轻量排查入口；
禁用作为状态和操作记录嵌入，不单独成模块；
阶段目标按 P0 信息标准化、P1 辅助排查、P2 深度治理、P3 主动预警逐步推进。
```
